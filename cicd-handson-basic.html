基礎編：自動テスト実行

GitHub Actionsでテストを自動化しよう

何を作るか？

🎯 目標

コードをGitHubにプッシュしたら、
自動的にPHPUnitが実行されるようにします。

実現すること

  • • コードをプッシュすると自動でテストが実行される
  • • GitHub上でテスト結果が見える
  • • テストが失敗したらすぐに通知される
  • • チーム全員のコードが常にテストされる

💡 前提条件

  • • TDDハンズオンでTodoアプリを作成済み
  • • GitHubアカウントを持っている
  • • Gitの基本操作ができる

ステップ1：GitHubリポジトリ作成

コードをGitHubにアップロードする

📁 新しいリポジトリを作成

  1. 1. GitHubで新規リポジトリ作成

    GitHub (https://github.com) にアクセスし、右上の「+」→「New repository」をクリック

    リポジトリ名: laravel-tdd-cicd（または任意の名前）

    公開設定: Public（無料プランでActions使用可能）

    README: チェックなし（既存プロジェクトなので）

  2. 2. ローカルリポジトリと連携

    Laravelプロジェクトのディレクトリで以下のコマンドを実行: 

    # まだGit管理していない場合
git init

# リモートリポジトリを追加（URLは自分のものに置き換える）
git remote add origin https://github.com/YOUR_USERNAME/laravel-tdd-cicd.git

# 現在のブランチ名をmainに変更
git branch -M main

  3. 3. 最初のコミット&プッシュ

    # 全てのファイルをステージング
git add .

# コミット
git commit -m "Initial commit with TDD Todo app"

# GitHubにプッシュ
git push -u origin main

✅ 確認: GitHubのリポジトリページをブラウザで開き、コードがアップロードされていることを確認してください。

ステップ2：ワークフローファイル作成

GitHub Actionsの設定を書く

📝 test.ymlを作成

Gitリポジトリのルート（.gitディレクトリがある場所）に以下のディレクトリ構造を作成します: 

リポジトリルート/    ← .gitディレクトリがある場所
├── .git/           ← Gitの管理ディレクトリ
├── .github/        ← ここを作る
│   └── workflows/
│       └── test.yml  ← これを作る
├── src/            ← Laravelのソースコード（プロジェクトによる）
│   ├── app/
│   ├── tests/
│   └── ...
└── docker-compose.yml

⚠️ 注意: .githubディレクトリは、Laravelプロジェクト（appやtestsがある場所）ではなく、 Gitリポジトリのルート（.gitディレクトリと同じ階層）に配置してください。

.github/workflows/test.yml を作成し、以下の内容を記述します: 

# .github/workflows/test.yml
name: Run Tests

# いつ実行するか
on:
  push:
    branches: [ main, master, develop ]
  pull_request:
    branches: [ main, master, develop ]

# 実行する処理
jobs:
  test:
    runs-on: ubuntu-latest

    steps:
    # 1. コードをチェックアウト
    - name: Checkout code
      uses: actions/checkout@v3

    # 2. PHPをセットアップ
    - name: Setup PHP
      uses: shivammathur/setup-php@v2
      with:
        php-version: '8.2'
        extensions: mbstring, pdo, pdo_sqlite
        coverage: none

    # 3. Composerの依存関係をインストール
    - name: Install dependencies
      working-directory: src
      run: composer install --prefer-dist --no-progress

    # 4. .envファイルをコピー
    - name: Copy .env
      working-directory: src
      run: cp .env.example .env

    # 5. アプリケーションキーを生成
    - name: Generate app key
      working-directory: src
      run: php artisan key:generate

    # 6. ディレクトリ権限を設定
    - name: Set directory permissions
      working-directory: src
      run: chmod -R 777 storage bootstrap/cache

    # 7. テストを実行
    - name: Run tests
      working-directory: src
      run: php artisan test

📖 ワークフロー構文の解説:

YAMLの階層構造

階層 何を定義？ キーワード例
1階層目 ワークフローの設定 name, on, jobs
2階層目 ジョブの定義 test:（ジョブ名）
3階層目 ジョブの設定 runs-on, steps
4階層目 個々のステップ - name:, uses:, run: 
# 1階層目：ワークフローの設定
name: Run Tests
on: [push, pull_request]
jobs:

  # 2階層目：ジョブの定義（ジョブ名）
  test:

    # 3階層目：ジョブの設定
    runs-on: ubuntu-latest
    steps:

      # 4階層目：個々のステップ
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Run tests
        run: php artisan test

YAMLの重要ポイント

インデント（字下げ）が重要

YAMLはインデントで階層を表現します。スペースの数がずれるとエラーになります。

  • ・スペース2個ずつ下げるのが一般的
  • ・タブ文字は使わない（スペースのみ）
  • ・同じ階層は同じインデント数にする

ジョブは複数追加できる

jobs: の下にジョブを追加すると、並列で実行されます。 

jobs:
  test:       # ジョブ1：テスト
    runs-on: ubuntu-latest
    steps: ...

  build:      # ジョブ2：ビルド（追加）
    runs-on: ubuntu-latest
    steps: ...

  deploy:     # ジョブ3：デプロイ（追加）
    runs-on: ubuntu-latest
    steps: ...

配列の書き方（- と []）

YAMLで配列（複数の値）を書く方法は2つあります。どちらも同じ意味です。 

# 書き方1：[] で横に並べる
branches: [main, master, develop]
# 書き方2：- で縦に並べる
branches:
  - main
  - master
  - develop

steps: の下の - name: も配列の要素です（ステップが複数あるため）

主なキーワード

name:

ワークフローやステップの名前（UIに表示）

on:

実行トリガー（push、pull_requestなど）

runs-on:

実行環境（ubuntu-latest など）

steps:

ステップの配列（上から順に実行）

uses:

外部アクションを使う

run:

シェルコマンドを実行

with:

アクションにパラメータを渡す

working-directory:

コマンド実行ディレクトリを指定

バージョン指定（@v3 など）

uses: actions/checkout@v3 の @v3 はバージョン指定です。

  • @v3 - メジャーバージョン3の最新（推奨）
  • @v3.1.0 - 特定のバージョンを固定
  • @main - ブランチを指定（非推奨）

📖 使用しているアクションの解説:

actions/checkout@v3

GitHub公式アクション。リポジトリのコードをダウンロードします。ほぼ全てのワークフローで最初に使います。

shivammathur/setup-php@v2

PHPをセットアップするアクション。バージョンや拡張機能をwith:で指定できます。

🔍 アクションの探し方:

GitHub Marketplace

公式のアクション検索サイトで、様々なアクションを探せます。

https://github.com/marketplace?type=actions

検索のコツ

  • 「github actions PHP」「github actions Laravel」などで検索
  • スター数や使用数が多いものを選ぶと安心
  • actions/で始まるものはGitHub公式

よく使うアクションの例：

  • actions/checkout - コードのチェックアウト
  • actions/cache - キャッシュの保存・復元
  • actions/upload-artifact - ファイルの保存
  • shivammathur/setup-php - PHPのセットアップ
  • actions/setup-node - Node.jsのセットアップ

📝 ワークフローファイルの作り方:

GitHubが自動で認識する仕組み

GitHubは .github/workflows/ ディレクトリ内の .yml ファイルを自動的にワークフローとして認識します。

  • ・特別な設定や登録は不要
  • ・ファイルをプッシュするだけで有効になる
  • ・ファイルを削除するとワークフローも消える

方法1：ローカルで作成（今回の方法）

  1. .github/workflows/ ディレクトリを作成
  2. その中に .yml ファイルを作成
  3. GitHubにプッシュすると自動で認識される

方法2：GitHub上で作成

  1. リポジトリの「Actions」タブをクリック
  2. 「New workflow」をクリック
  3. テンプレートを選ぶか、「set up a workflow yourself」で一から作成

ファイル名について

  • ファイル名は自由（例：test.yml、ci.yml、deploy.yml）
  • 複数のファイルを作ると、複数のワークフローが動く
  • 拡張子は .yml または .yaml

💡 実務でのおすすめ：AIツールで作成

実際の開発現場では、プロジェクトディレクトリでCLIのAIツール（Claude Code、GitHub Copilot CLIなど）を開いて、やりたいことをプロンプトで伝えて作成させる方が早く、ほとんど間違えません。

プロンプト例：

「このLaravelプロジェクト用のGitHub Actionsを作成して。pushとPR時にPHPUnitを実行したい」

AIがプロジェクト構造を読み取り、適切なワークフローファイルを生成してくれます。

💡 プロジェクト構造について:
上記の例は、Laravelプロジェクトがsrcディレクトリ内にある構造を想定しています。 もしLaravelプロジェクトがリポジトリのルートに直接ある場合（composer.jsonがルートにある場合）は、 working-directory: srcの行を削除してください。

🔄 テスト時にマイグレーションを自動実行する

⚠️ TDDハンズオンでDB版Todoを実装した場合、このエラーが発生します

FAIL  Tests\Feature\ExampleTest
⨯ the application returns a successful response

PDOException: SQLSTATE[HY000]: General error: 1 no such table: todos

🤔 なぜこのエラーが起きるのか？

テストコード（ExampleTest.php）自体はtodosテーブルを直接使っていません。
しかし、テストがアクセスするページの処理がデータベースを使っています。 

ExampleTest
    ↓ $this->get('/')
ルート "/"
    ↓ TodoController@index
TodoController::index()
    ↓ Todo::orderBy(...)->get()
Todoモデル
    ↓ SELECT * FROM todos
❌ todosテーブルが存在しない！

テスト用のSQLiteデータベースは空の状態で作られるため、マイグレーションを実行してテーブルを作成する必要があります。

✅ 解決方法：RefreshDatabase トレイトを使う

LaravelのRefreshDatabaseトレイトを使うと、
各テスト実行時にマイグレーションが自動実行されます。

tests/Feature/ExampleTest.php を以下のように修正してください： 

<?php

namespace Tests\Feature;

use Illuminate\Foundation\Testing\RefreshDatabase;  // ← コメントを外す
use Tests\TestCase;

class ExampleTest extends TestCase
{
    use RefreshDatabase;  // ← これを追加

    public function test_the_application_returns_a_successful_response(): void
    {
        $response = $this->get('/');
        $response->assertStatus(200);
    }
}

📖 RefreshDatabase の動作

  • 各テスト実行前 - マイグレーションを実行してテーブルを作成
  • 各テスト実行後 - データベースをリセット（他のテストに影響しない）
  • インメモリSQLite - 高速に動作（ファイルに書き込まない）

💡 ポイント：データベースを使うFeatureテストにはuse RefreshDatabase;を追加してください。
データベースを使わないUnitテスト（Calculatorなど）には不要です。

📝 phpunit.xml の設定確認

CI環境ではMySQLが使えないため、phpunit.xmlでSQLiteを使うよう設定します。
以下の設定が有効になっていることを確認してください。 

<!-- phpunit.xml -->
<php>
    <env name="DB_CONNECTION" value="sqlite" force="true"/>
    <env name="DB_DATABASE" value=":memory:" force="true"/>
</php>

📖 設定の意味：

  • DB_CONNECTION="sqlite" - SQLiteを使用（CI環境で利用可能）
  • DB_DATABASE=":memory:" - メモリ上にDBを作成（高速・テスト後に自動削除）
  • force="true" - .envの設定を上書きしてこの値を使用

ステップ3：プッシュして動作確認

実際にCIを動かしてみる

🚀 ワークフローを実行

  1. 1. 作成したファイルをコミット&プッシュ

    git add .github/workflows/test.yml
git commit -m "Add GitHub Actions workflow for testing"
git push

  2. 2. GitHubでActionsタブを確認

    GitHubのリポジトリページで「Actions」タブをクリックします。

    「Run Tests」というワークフローが実行中になっているはずです。

    🟡 黄色（実行中） → 🟢 緑（成功） または 🔴 赤（失敗）

  3. 3. 実行ログを確認

    ワークフロー名をクリックすると、詳細なログが見られます。

    • • 各ステップの実行時間
    • • コマンドの出力結果
    • • テストの成功/失敗
    • • エラーメッセージ（失敗時）

✅ 成功した場合

おめでとうございます！自動テストが動きました🎉
これで、コードをプッシュするたびに自動でテストが実行されます。

❌ 失敗した場合

ログを確認して、エラーメッセージを読みましょう。
よくある原因:

  • • PHPのバージョンが合っていない
  • • 必要な拡張機能がインストールされていない
  • • テストコード自体にバグがある
  • • .envの設定が不足している

応用：ステータスバッジを追加

READMEにテスト結果を表示する

🏅 バッジで見える化

GitHubのステータスバッジをREADMEに追加すると、一目でテスト状態が分かります。
オープンソースプロジェクトでよく使われる機能です。

バッジの表示例:

🟢 passing （テストが成功している場合）

🔴 failing （テストが失敗している場合）

README.md の先頭に以下を追加: 

# Laravel TDD CICD

![Run Tests](https://github.com/YOUR_USERNAME/YOUR_REPO/workflows/Run%20Tests/badge.svg)

## 概要
TDDで開発したTodoアプリです。
CI/CDでテストを自動化しています。

💡 ポイント: YOUR_USERNAMEとYOUR_REPOを自分のものに置き換えてください。

重要：テスト失敗時にマージを防ぐ

ブランチ保護ルールでコード品質を守る

⚠️ 現状の問題

今のままでは、テストが失敗してもpushできてしまいます。

GitHub Actionsはリモートで実行されるだけなので、ローカルからのpushを止めることはできません。
これを防ぐために、プルリクエスト + ブランチ保護ルールを使います。

🛡️ ブランチ保護ルールとは

✅ プルリクエスト + ブランチ保護ルールが圧倒的に一般的です。

チーム開発の標準手法として、世界中の開発チームで使われています。

ブランチ保護ルールは、mainブランチに直接pushするのではなく、
必ずプルリクエスト（PR）を経由してマージする仕組みです。

スクラム開発での位置づけ

プルリクエストは、スクラムの「完成の定義（Definition of Done）」を技術的に保証する仕組みです。

  • ✓ コードレビューが承認されている
  • ✓ すべてのテストがパスしている（CI）
  • ✓ コード品質基準を満たしている
  • ✓ チーム全員が変更内容を把握できる（透明性）

📋 設定手順

🤔 なぜブランチ保護ルールを設定するのか？

GitHub Actionsでテストを自動実行しても、それだけでは「テストが失敗しても普通にpushできてしまう」状態です。

ブランチ保護ルールを設定することで：

  • • mainブランチへの直接pushを禁止 → 必ずPRを経由
  • • テストが通らないとマージ不可 → 壊れたコードがmainに入らない
  • • チーム全員に同じルールを強制 → 「うっかりpush」を防止
  1. 1. GitHubでブランチ保護ルールを設定
    1. GitHubのリポジトリページで「Settings」タブをクリック
    2. 左メニューから「Code and automation」→「Rules」→「Rulesets」を選択
    3. 「Add branch ruleset」ボタンをクリック（緑色のボタン）
    4. 「Ruleset Name」に任意の名前（例：Protect production branches）を入力
    5. 「Enforcement status」をActiveに設定
    6. 「Target branches」で「Add target」をクリックし、「Include by pattern」を選択
    7. ブランチ名にmainと入力して追加

      💡 ヒント: プロジェクトによって本番ブランチ名（main、master、productionなど）が異なります。
      自分のプロジェクトのブランチ名を確認してください。

    8. 「Branch rules」セクションで以下をチェック（詳しい説明は下記参照）
    9. ページ下部の「Create」ボタンをクリック
  2. 2. 各設定項目の意味
    ☑️

    Require a pull request before merging

    意味：mainブランチに直接pushできなくなり、必ずプルリクエスト（PR）を経由してマージする必要があります。

    なぜ必要？

    PRを経由することで、コードレビューの機会が生まれ、GitHub Actionsのチェックを必ず通すことができます。 「うっかりmainにpushしてしまった」という事故を防ぎます。

    追加オプション：Required approvals

    マージ前に必要なレビュー承認の数。0だとレビューなしでマージ可能、1以上だと誰かの承認が必要。
    チーム開発では1以上を推奨（コードレビュー文化を作るため）。

    ☑️

    Require status checks to pass

    意味：指定したチェック（テストなど）が成功しないとマージできなくなります。

    なぜ必要？

    これがCI/CDの要です。テストが失敗したコードをmainにマージすることを物理的に防ぎます。 「テスト失敗してるけど急いでるからマージしちゃえ」ができなくなります。

    ⚠️ 重要：チェックを追加しないと意味がない！

    このオプションをONにしただけでは不十分です。
    「+ Add checks」をクリックして、どのチェックを必須にするか指定する必要があります。

    ➕

    + Add checks（チェックの追加）

    意味：マージ前に必ず成功しなければならないチェック（ワークフロー）を指定します。

    設定方法

    1. 「+ Add checks」ボタンをクリック
    2. 検索ボックスに「test」と入力
    3. 表示された「test」（GitHub Actionsのジョブ名）を選択

    ※ワークフローを一度も実行していないと検索に表示されません。先にpushしてActionsを動かしてください。

    💡 ジョブ名はどこで決まる？

    test.ymlのjobs:の下に書いた名前（今回はtest:）がチェック名になります。

  3. 3. 今後の開発フロー
    # 1. 新しいブランチを作成
git checkout -b feature/new-feature

# 2. TDDで機能開発
# ... コードを書く ...

# 3. コミット&プッシュ
git add .
git commit -m "Add new feature"
git push origin feature/new-feature

# 4. GitHubでプルリクエストを作成
# ブラウザでGitHubを開き「Compare & pull request」ボタンをクリック
  4. 4. プルリクエストでの確認事項

    PRを作成すると、自動的に以下が実行されます：

    • 🟡 GitHub Actionsでテストが実行される
    • 🟢 テストが成功 → 「Merge pull request」ボタンが有効になる
    • 🔴 テストが失敗 → マージできない（修正が必要）

    ✅ これで、テストが通らないコードはmainブランチに入らなくなります！

💡 他の方法との比較

方法 使用頻度 メリット デメリット
PR + ブランチ保護 ⭐⭐⭐ 非常に一般的 • チーム全体で強制可能
• コードレビュー文化が育つ
• GitHubの標準機能
• 変更履歴が明確 		• PRの手間が増える
• 小規模な変更も同じ手順
Git Hooks (pre-push) ⭐ 補助的に使用 • ローカルで即座にチェック
• 個人開発に便利 		• 簡単にスキップ可能
• チーム全体への適用が困難
• 各自がセットアップ必要

推奨： スクラム開発では、PR + ブランチ保護ルールを必ず設定しましょう。
これにより、スプリント内でのコード品質が保たれ、技術的負債の蓄積を防ぐことができます。

👥 チームワーク

ワークフローを拡張してみよう！

基本的なテスト自動化ができました。
以下の演習で、より実践的なCI/CDパイプラインを構築してみましょう。

📝 演習1：ジョブ名を変更する

現在のジョブ名 test を、より分かりやすい名前に変更してみましょう。 

jobs:
  phpunit-tests:    # ← 名前を変更
    runs-on: ubuntu-latest

💡 変更後は、ブランチ保護ルールの「Add checks」も更新が必要です。

🔢 演習2：複数のPHPバージョンでテストする

マトリックスビルドを使って、PHP 8.2 と 8.3 の両方でテストを実行してみましょう。

📍 変更箇所1：runs-on の下に追加

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:                          # ← この3行を追加
      matrix:
        php-version: ['8.2', '8.3']

    steps:

📍 変更箇所2：Setup PHP の php-version を変更

    - name: Setup PHP
      uses: shivammathur/setup-php@v2
      with:
        php-version: '8.2'                        # 削除
        php-version: ${{ matrix.php-version }}   # 追加
        extensions: mbstring, pdo, pdo_sqlite

✅ 確認方法

pushした後、GitHub ActionsのUIで2つのジョブ（PHP 8.2 と 8.3）が並列で実行されることを確認してください。

質問：どういう時に使う？（クリックで展開）

よく使うケース：

  • オープンソースのライブラリ/パッケージ開発
  • 複数のPHPバージョンをサポートする必要がある場合
  • PHPのバージョンアップを検討している時の事前検証

あまり使わないケース：

  • 一般的なWebアプリ開発（本番環境のPHPバージョンは1つだけ）
  • 社内システム開発

⚡ 演習3：Composerキャッシュを追加する

依存関係のインストールを高速化するため、キャッシュを追加してみましょう。

📍 変更箇所：Install dependencies の直前に追加

    - name: Set directory permissions
      working-directory: src
      run: chmod -R 777 storage bootstrap/cache

    # ↓↓↓ ここから追加 ↓↓↓
    - name: Cache Composer dependencies
      uses: actions/cache@v3
      with:
        path: src/vendor
        key: ${{ runner.os }}-composer-${{ hashFiles('src/composer.lock') }}
        restore-keys: |
          ${{ runner.os }}-composer-
    # ↑↑↑ ここまで追加 ↑↑↑

    - name: Install dependencies
      working-directory: src
      run: composer install --prefer-dist --no-progress

✅ 確認方法

1回目の実行ログで「Cache not found」、2回目以降で「Cache restored successfully」と表示されれば成功です。

節約効果の確認：

Install dependencies ステップの実行時間を比較してください。

キャッシュなし（1回目） 30秒〜1分以上
キャッシュあり（2回目以降） 数秒〜10秒程度

※ 依存関係が多いプロジェクトほど効果が大きくなります

質問：どういう時に使う？（クリックで展開）

よく使うケース：

  • 依存関係が多いプロジェクト（composer installに時間がかかる）
  • CIの実行時間を短縮したい場合
  • GitHub Actionsの無料枠を節約したい場合

補足：

最近の shivammathur/setup-php にはキャッシュ機能が内蔵されているため、手動設定が不要な場合もあります。
