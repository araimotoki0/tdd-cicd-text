CI/CD: スクラム開発を加速させる自動化

継続的インテグレーション・継続的デリバリーでスプリントの価値を最大化

スプリント内で品質を保ちながら、プロダクトオーナーに迅速にフィードバック

Code → Build → Test → Deploy

CI/CDとは？

📘 スクラム開発におけるCI/CD

CI/CDは、スクラムの価値観である「透明性」「検査」「適応」を 技術的に実現する自動化のプラクティスです。
スプリント内でコードの変更を自動的にテスト・ビルド・デプロイすることで、 プロダクトオーナーやステークホルダーに迅速にフィードバックを提供できます。

🔄 CI: Continuous Integration

継続的インテグレーション

  • • デイリーで開発チームのコードを統合
  • • 自動テストでインクリメントの品質を確認
  • • スプリント中の統合リスクを削減
  • • デイリースクラムで進捗を透明化

🚀 CD: Continuous Delivery/Deployment

継続的デリバリー/デプロイ

  • • スプリントレビューで動くソフトウェアをデモ
  • • プロダクトバックログ項目を迅速にリリース
  • • ステークホルダーへの早期フィードバック
  • • Done（完成）の定義を技術的に保証

💡 スクラムとの親和性: CI/CDは自動化により、スクラムチームが 「動くソフトウェア」を毎スプリント提供することを技術的に支援します。 プロダクトオーナーは常に最新のインクリメントを確認でき、優先順位の判断が迅速になります。

🤔 スクラム開発でCI/CDが必要な理由

❌ CI/CDがないスクラムチームの課題

  • • スプリント終了時の統合で問題が発覚 → スプリントゴール達成が危うい
  • • 手動デプロイに時間がかかる → スプリントレビューに間に合わない
  • • テストを省略 → Done（完成）の定義を満たせない
  • • 環境構築に時間を取られる → 本来の開発に集中できない
  • • プロダクトオーナーが実物を確認できない → フィードバックが遅れる

✅ CI/CDがあるスクラムチームの強み

  • • デイリースクラムで常に統合済みの状態を確認
  • • スプリントレビューで自信を持ってデモできる
  • • プロダクトオーナーが毎日進捗を確認できる
  • • 技術的負債が蓄積しにくい
  • • Velocity（開発速度）が安定・向上する

🔧 スプリント内でのCI/CDパイプライン

1

📝 Code（実装）

スプリントバックログ項目を実装し、GitHubにプッシュします。

→ デイリースクラムで進捗を共有

2

🔨 Build（ビルド）

依存関係のインストール、アセットのコンパイルなどを自動実行します。

→ 環境差異によるトラブルを防止

3

🧪 Test（テスト）

PHPUnit、PHPStanなどを自動実行してDone（完成）の定義を保証します。

→ 品質基準を満たしていることを確認

4

🚀 Deploy（デプロイ）

ステージング環境へ自動デプロイし、スプリントレビューで実演します。

→ プロダクトオーナー・ステークホルダーへのフィードバック

💡 スクラムでのポイント: これら全てが自動的に実行されるため、 開発チームは価値提供に集中でき、プロダクトオーナーは透明性のあるインクリメントを常に確認できます。

GitHub Actionsとは？

🤖 GitHubが提供するCI/CDツール

GitHub Actionsは、GitHubに統合されたCI/CDプラットフォームです。
リポジトリ内に設定ファイルを置くだけで、様々な自動化が可能になります。

✅ メリット

  • • GitHubと完全統合
  • • 無料枠が豊富
  • • 設定が簡単
  • • 豊富なアクション（再利用可能な部品）
  • • コミュニティが活発

🎯 できること

  • • 自動テスト実行
  • • コード品質チェック
  • • 自動デプロイ
  • • Slack通知
  • • その他あらゆる自動化
# .github/workflows/test.yml
name: Run Tests

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Setup PHP
        uses: shivammathur/setup-php@v2
        with:
          php-version: '8.2'
      - name: Install Dependencies
        run: composer install
      - name: Run Tests
        run: php artisan test

↑ この設定ファイル1つで、コードをプッシュするたびに自動でテストが実行されます！

📚 GitHub Actionsの主要概念

Workflow（ワークフロー）

自動化の全体の流れ。YAMLファイルで定義します。
例：「テストを実行するワークフロー」「デプロイするワークフロー」

Job（ジョブ）

ワークフローを構成する作業単位。複数のジョブを並列実行できます。
例：「testジョブ」「deployジョブ」

Step（ステップ）

ジョブを構成する個別の作業。順番に実行されます。
例：「コードをチェックアウト」「PHPをセットアップ」「テスト実行」

Action（アクション）

再利用可能な部品。コミュニティが作成したものを利用できます。
例：actions/checkout@v3、shivammathur/setup-php@v2

スクラムイベントとCI/CD

🔗 各スクラムイベントでCI/CDが果たす役割

📅 デイリースクラム（Daily Scrum）

CI/CDにより、毎日の統合状態を透明化できます。

  • ✅ 昨日プッシュしたコードが自動テストでグリーン
  • ✅ 統合の問題を15分の会議前に把握
  • ✅ 「今日やること」が技術的に実現可能か確認済み
  • ✅ チーム全体の進捗がGitHubのダッシュボードで可視化

🎯 スプリントレビュー（Sprint Review）

CI/CDで自動デプロイされた環境でデモを実施できます。

  • ✅ 動くソフトウェアをステージング環境でデモ
  • ✅ プロダクトオーナーから即座にフィードバックを受ける
  • ✅ Done（完成）の定義が技術的に保証されている
  • ✅ ステークホルダーが実際に触れる環境を提供

🔄 スプリントレトロスペクティブ（Sprint Retrospective）

CI/CDパイプラインの改善もレトロで議論します。

  • ✅ テストの実行時間を短縮できないか検討
  • ✅ デプロイの頻度を上げる方法を議論
  • ✅ 技術的負債の蓄積を振り返る
  • ✅ CI/CD改善が次スプリントのバックログに

📋 スプリントプランニング（Sprint Planning）

CI/CDがあることで、見積もりの精度が向上します。

  • ✅ 環境構築やデプロイの時間を考慮不要
  • ✅ Done（完成）の定義が明確で共有されている
  • ✅ 技術的リスクが低い状態でプランニング
  • ✅ Velocity（速度）が安定している

💡 スクラム開発での標準的なフロー

  1. スプリントバックログ項目をTDDで実装
  2. GitHubにプッシュ
  3. CI/CDが自動でテスト・品質チェック
  4. プルリクエストをレビュー・マージ
  5. 自動デプロイでステージング環境へ
  6. デイリースクラムで進捗を共有
  7. スプリントレビューでデモ・フィードバック

📜 Done（完成）の定義にCI/CDを組み込む

CI/CDを活用することで、Done（完成）の定義を技術的に保証できます。

Done（完成）の定義例

  • ☑ コードレビューが承認されている
  • ☑ すべての自動テストがパスしている（CI）
  • ☑ コード品質チェックをクリア（CI）
  • ☑ ステージング環境へ自動デプロイ済み（CD）
  • ☑ プロダクトオーナーが承認している
  • ☑ ドキュメントが更新されている
