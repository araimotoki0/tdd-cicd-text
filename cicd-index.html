CI/CD: Continuous Integration / Continuous Delivery

継続的インテグレーション・継続的デリバリーで開発を加速させよう

テストを自動化し、品質を保ちながら高速にリリースする

Code → Build → Test → Deploy

CI/CDとは？

📘 正式な定義

CI/CDは、ソフトウェア開発における自動化のプラクティスです。
コードの変更を自動的にテスト・ビルド・デプロイすることで、 品質を保ちながら高速にリリースできます。

🔄 CI: Continuous Integration

継続的インテグレーション

  • • コードを頻繁にマージする
  • • 自動テストで品質を確認
  • • 問題を早期に発見
  • • チーム全体の生産性向上

🚀 CD: Continuous Delivery/Deployment

継続的デリバリー/デプロイ

  • • いつでもリリース可能な状態を保つ
  • • デプロイプロセスを自動化
  • • 本番環境への安全なリリース
  • • フィードバックサイクルの高速化

💡 重要: CI/CDは自動化がキーワードです。 人が手作業でやっていたテスト・ビルド・デプロイを自動化することで、 ミスを減らし、開発スピードを向上させます。

🤔 なぜCI/CDが必要なのか？

❌ CI/CDがない場合の問題

  • • テストを手動で実行するのが面倒 → テストが省略される
  • • デプロイ手順が複雑 → リリースが遅れる
  • • 環境差異で本番で動かない → デプロイが怖い
  • • 複数人の変更が衝突 → 統合に時間がかかる
  • • バグが本番で発覚 → ユーザーに迷惑をかける

✅ CI/CDがある場合のメリット

  • • コードをプッシュすると自動でテスト実行
  • • テストが通れば自動でデプロイ
  • • 問題をすぐに発見できる
  • • 安心してリリースできる
  • • チームの開発速度が向上

🔧 CI/CDパイプラインの流れ

1

📝 Code（コード作成）

開発者がコードを書き、GitHubにプッシュします。

2

🔨 Build（ビルド）

依存関係のインストール、アセットのコンパイルなどを自動実行します。

3

🧪 Test（テスト）

PHPUnit、PHPStanなどを自動実行して品質をチェックします。

4

🚀 Deploy（デプロイ）

テストが成功したら、ステージングや本番環境へ自動デプロイします。

💡 ポイント: これら全てが自動的に実行されます。 開発者は「コードを書いてプッシュする」だけで、残りは自動化されています。

GitHub Actionsとは？

🤖 GitHubが提供するCI/CDツール

GitHub Actionsは、GitHubに統合されたCI/CDプラットフォームです。
リポジトリ内に設定ファイルを置くだけで、様々な自動化が可能になります。

✅ メリット

  • • GitHubと完全統合
  • • 無料枠が豊富
  • • 設定が簡単
  • • 豊富なアクション（再利用可能な部品）
  • • コミュニティが活発

🎯 できること

  • • 自動テスト実行
  • • コード品質チェック
  • • 自動デプロイ
  • • Slack通知
  • • その他あらゆる自動化
# .github/workflows/test.yml
name: Run Tests

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Setup PHP
        uses: shivammathur/setup-php@v2
        with:
          php-version: '8.2'
      - name: Install Dependencies
        run: composer install
      - name: Run Tests
        run: php artisan test

↑ この設定ファイル1つで、コードをプッシュするたびに自動でテストが実行されます！

📚 GitHub Actionsの主要概念

Workflow（ワークフロー）

自動化の全体の流れ。YAMLファイルで定義します。
例：「テストを実行するワークフロー」「デプロイするワークフロー」

Job（ジョブ）

ワークフローを構成する作業単位。複数のジョブを並列実行できます。
例：「testジョブ」「deployジョブ」

Step（ステップ）

ジョブを構成する個別の作業。順番に実行されます。
例：「コードをチェックアウト」「PHPをセットアップ」「テスト実行」

Action（アクション）

再利用可能な部品。コミュニティが作成したものを利用できます。
例：actions/checkout@v3、shivammathur/setup-php@v2

TDDとCI/CDの関係

🔗 相乗効果を生み出す組み合わせ

TDDで作ったテストが、CI/CDで活きる

TDDで書いたテストコードは、CI/CDで自動実行されることで真価を発揮します。

✅ コードをプッシュするたびに自動でテスト実行

✅ バグを即座に検知

✅ チーム全員の変更が常にテストされる

✅ 安心してリファクタリングできる

CI/CDがあると、TDDがもっと楽しくなる

自動化により、テストを書くモチベーションが上がります。

✅ テスト実行が自動化されて楽

✅ GitHub上でテスト結果が見える化

✅ プルリクエストで品質が保証される

✅ チーム全体で品質意識が高まる

💡 実務での標準的な開発フロー

  1. TDDでテストを書きながら機能を実装
  2. GitHubにプッシュ
  3. CI/CDが自動でテスト実行
  4. テストが通ればプルリクエストをマージ
  5. 自動デプロイでステージング/本番環境へ
