## 基礎編の議論

### 議論0（ウォームアップ）：自社と他社の開発フローを共有しよう

#### ねらい
本題（議論1〜3）に入る前のアイスブレイク兼・現状把握。受講者それぞれの「コードを書いてから本番に届くまで」を言語化させ、**他社と比べて自社を相対化**させるのが目的。ここで出た“自社の現状”が、後続の議論の具体的な土台になる。

#### 進め方
1. 1人30秒〜1分で「ブランチ運用 / テスト / レビュー / リリース頻度 / デプロイ」を共有
2. 全員が話したら「一番自動化が進んでいる会社」「一番手作業が多い会社」を比べる
3. その差がどこから来ているか（チーム規模・業界・歴史的経緯）を軽く掘る

#### 現場でよくあるフローの型（講師が例として出すと議論が回る）
- **スタートアップ / 自社開発**: ブランチ＋PR、CIで自動テスト、mainマージで自動デプロイ。1日に何度もリリース。「とにかく速く回す」
- **大企業 / SIer**: ブランチ運用は厳格、レビュー必須、ステージング経由でリリース。リリースは週次〜月次で「承認フロー」が重い
- **受託 / SES**: 顧客のルールに従う。CIがある現場とゼロの現場が混在。「客先によって全然違う」が口癖になりがち
- **レガシー現場**: mainに直push、テストなし、サーバーに手でアップロード。「動いてるから触らない」

#### 驚きポイント
> 「正解のフロー」は存在しない。**会社の規模・業界・プロダクトの性質で最適解が変わる。**
>
> 大事なのは「他社が進んでいる/遅れている」というジャッジではなく、
> **「なぜそのフローになっているのか」の理由**に気づくこと。
> 速く回す会社は速さに、堅い会社は堅さに、それぞれ理由がある。

#### ファシリのコツ
- 「自社にCIが無い」人が萎縮しないように、「無いのが普通の現場も多い。だから今日学ぶ価値がある」とフォローする
- 出てきたフローを軽くホワイトボード/メモに並べると、議論1〜3で「さっきの〇〇さんの会社の話だと…」と繋げられる

### 議論1：完成の定義（DoD）にCI/CDを追加するか？

#### 結論から言うと
**「CIが通ること」はDoDに入れるべき。ただし「通らないとマージできない」という仕組みで強制する方がベター。**

#### 驚きポイント
> 「DoDに書く」と「仕組みで強制する」は違います。
>
> DoDに「CIが通ること」と書いても、人間は忘れます。急いでいると「今回だけ...」と例外を作ります。
>
> **本当にやるべきことは「ブランチ保護ルール」で物理的にマージできなくすること。**
> DoDは「なぜそのルールがあるのか」をチームが理解するためのドキュメントです。

#### 実際の現場では
- **スタートアップ**: 「スピード優先でCI通らなくてもマージしてた。結果、本番障害が頻発して、結局CI必須にした」
- **大企業**: 「CI必須は当然。むしろCIに30分かかるのが問題で、高速化が課題」
- **受託開発**: 「顧客によってルールが違う。厳しい顧客だと、CI + コードレビュー + セキュリティスキャン全部必須」

### 議論2：テストが失敗した時のチーム対応ルール

#### 結論から言うと
**「壊した人が直す」ではなく「チーム全員の問題」として扱う。ただし、最初に気づいた人がまず声を上げる文化が大事。**

#### 驚きポイント
> Googleには「誰がCIを壊しても、最初に気づいた人が直す」という文化があります。
>
> なぜか？**「犯人探し」を始めると、みんなコミットを怖がるようになるから。**
>
> テストが壊れた時に「誰だよ壊したの」と言うチームより、「あ、壊れてる。俺が見とくわ」と言えるチームの方が強い。

#### 実際の現場では
- **アンチパターン**: 「壊した人を Slack で晒す」→ コミットが減る、隠蔽体質になる
- **ベストプラクティス**: 「CI失敗したら全員に通知、手が空いてる人が対応」→ 心理的安全性が高まる
- **現実的な落とし所**: 「まず壊した人に連絡、15分反応なければ誰かが引き取る」

#### Slackでの通知例
```
🔴 CI Failed: main branch
最後のコミット: @tanaka - "Add login feature"
誰か見れる人いますか？
```
「犯人」ではなく「情報」として共有するのがポイント。

### 議論3：ブランチ保護ルールをどこまで厳しくする？

#### 結論から言うと
**最初は緩く、問題が起きたら厳しくする。「予防」より「信頼」から始める。**

#### 驚きポイント
> 多くのチームが「最初から厳しくしすぎて」失敗します。
>
> - CI必須 ✅
> - レビュー1人必須 ✅
> - レビュー2人必須 → チームが3人なら回らない
> - 管理者もルール対象 → 緊急時に詰む
>
> **ルールは「過去に起きた問題」を防ぐために追加するもの。**
> まだ起きていない問題のために厳しくしすぎると、開発速度が落ちます。

#### 段階的に厳しくする例
1. **最初**: CI必須のみ
2. **レビュー漏れでバグが出た**: レビュー1人必須を追加
3. **レビューが形骸化した**: 「Approve後に追加コミット」を禁止
4. **本番障害が起きた**: 管理者も例外なしに変更

#### 緊急時のエスケープハッチ
```
厳しいルールを設定する場合は、必ず「緊急時の解除手順」も決めておく。

例：
- 本番障害時は、CTO承認でルール一時解除可能
- 解除した場合は、後日ポストモーテムで振り返る
```

## 応用編の議論

### 議論1：PHPStanのチェックレベルをどうする？

#### 結論から言うと
**新規プロジェクトはレベル5以上、既存プロジェクトはレベル1から始めて徐々に上げる。**

#### 驚きポイント
> PHPStanのレベルは「品質の高さ」ではなく「型情報の厳密さ」を表しています。
>
> - レベル1: 明らかなバグ（存在しないメソッド呼び出しなど）
> - レベル5: 型の不一致（intを期待する関数にstringを渡すなど）
> - レベル9: 完璧な型付け（mixedやanyを許さない）
>
> **レベル9が「良い」わけではない。** レベル9はTypeScriptの`strict: true`のようなもので、レガシーコードには現実的ではありません。

#### 現場でよくある失敗
```
「よし、品質を上げるぞ！」
→ いきなりレベル8を設定
→ 3000件のエラー
→ 「これ全部直すの無理...」
→ PHPStan導入中止
```

#### 成功パターン
```
1. レベル1で導入（エラー50件）
2. 50件を1週間で修正
3. レベル2に上げる（エラー30件追加）
4. 以下繰り返し...

重要：「レベルを上げる」ことをスプリントゴールに入れる
```

### 議論2：コードスタイルのルールをどこまで統一する？

#### 結論から言うと
**「自動で直せるものは全部自動で直す」が正解。人間が議論するのは時間の無駄。**

#### 驚きポイント
> コードスタイルの議論で最も生産性を下げるのは「人間が議論すること」です。
>
> - 「インデントはスペース2つ派」vs「4つ派」
> - 「波括弧は同じ行派」vs「次の行派」
>
> **これらの議論に正解はありません。どっちでもいい。**
>
> 大事なのは「チームで統一されていること」であって「どちらが正しいか」ではない。
> だから PSR-12 という標準規約をそのまま使うのがベスト。

#### 議論すべきこと・すべきでないこと
```
議論すべきでないこと（自動化で解決）：
- インデント幅
- 改行位置
- スペースの有無

議論すべきこと（自動化できない）：
- 変数名・関数名の命名規則
- ファイル・ディレクトリ構成
- コメントをどこまで書くか
```

#### 実践的なアドバイス
```php
// これはCS Fixerで自動修正される（議論不要）
if($x){return true;}
↓
if ($x) {
    return true;
}

// これは人間が決める必要がある（議論が必要）
$user_name  // スネークケース
$userName   // キャメルケース
```

### 議論3：品質チェックをDone of Definitionに入れる？

#### 結論から言うと
**入れる。ただし「CI/CDで自動チェック」という形で入れる。手動チェックリストにはしない。**

#### 驚きポイント
> DoDに「PHPStanエラー0」と書くのは良いですが、それを「人間が毎回確認する」のは間違いです。
>
> **確認作業は全てCIに任せる。人間は「CIが通ったかどうか」だけを見る。**
>
> なぜか？人間は必ずミスをするから。
> 「PHPStan通ったっけ？」「たぶん大丈夫」→ 本番障害

#### DoDの書き方
```
悪い例：
□ PHPStanを実行してエラーが0件であること
□ PHP CS Fixerを実行して警告が0件であること
□ テストが全て通ること

良い例：
□ CIが全てグリーンであること
  （CI内でPHPStan、CS Fixer、テストを自動実行）
```

#### さらに進んだチーム
```
最強のDoD：
□ mainブランチにマージされていること

なぜこれで十分か？
→ マージにはCIパス + レビュー承認が必須
→ CI内で全ての品質チェックが実行される
→ 「マージできた = 品質基準を満たしている」
```

## 議論のファシリテーションTips

### 沈黙が続いたら
- 「ちなみに、私が以前いたチームでは...」と自分の経験を話す
- 「〇〇さんの会社ではどうですか？」と指名する

### 議論が白熱しすぎたら
- 「どちらも正解ですね。大事なのはチームで合意することです」
- 「この議論、実は正解がないんです」と切り上げる

### 良い発言が出たら
- 「それ、すごく良い視点ですね」と具体的に褒める
- 「今の意見、他のチームにも共有したいですね」

### 最後のまとめで言うと良いこと
> 「今日議論したことに正解はありません。
> 大事なのは、**チームで話し合って決めること**です。
> 決めたルールがうまくいかなければ、また変えればいい。
> それがアジャイルです。」
